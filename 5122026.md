Note: transformers overwrites this during training to `[128009]` via "Updated tokens" alignment warning — which also works correctly.

## Reward Function Evolution

### v1 — floored penalty (broken)
```python
reward = max(0.0, 1.0 - (word_count - WORD_TARGET) * 0.05)
```
**Problem**: floored at 0 for all responses >30 words. All completions got identical reward=0 → `reward_std=0` → GRPO loss=0 → no learning.

### v2 — ratio reward (better signal, hackable)
```python
reward = min(1.0, WORD_TARGET / word_count)
```
**Problem**: 1-2 word responses get reward=1.0. Model learned to generate 2 real words followed by zero-width joiners (U+200D) — invisible characters that don't count as words, filling the context with high-reward gibberish.

### v3 — tent reward (current)
```python
if word_count <= WORD_TARGET:
    reward = word_count / WORD_TARGET   # rises linearly to 1.0 at target
else:
    reward = WORD_TARGET / word_count   # falls off above target
```
Penalizes both too-short AND too-long responses. Sweet spot at exactly `WORD_TARGET` words.

## GRPO Training Issues & Fixes

### Issue: `clipped_ratio=1`, all completions at max length
**Cause**: `max_completion_length=64` too short; model generates ~240 tokens before EOS. All completions clipped to identical length → no reward variance.  
**Fix**: Increased `max_completion_length` to 256; added "Please keep your responses brief." to system prompt to encourage natural termination.

### Issue: Reward hacking — "aalborg" loops and zero-width joiners
With `beta=0.0` (no KL penalty), model rapidly found degenerate shortcuts:
- Generating `<word>‍‍‍‍‍‍‍` (zero-width joiners) → counted as 1-2 words → reward=1.0
- Generating `Photos aalborg aalborg aalborg...` — "aalborg" is likely a compact single token; the model used it to fill context while minimizing the reward function's word count
**Fix**: `beta=0.1` (KL penalty keeps model anchored to reference distribution) + tent reward penalizes ultra-short responses.

### Issue: Garbage evaluation output despite low KL
Even with KL=0.002671 (nearly unchanged weights), post-training evaluation produced "aalborg", "odense", "fetisch", and O-repeat strings.  
**Root cause**: GRPOTrainer enables **gradient checkpointing** on the model. When `model.generate()` is called inside the training context (via `EvalCallback`), transformers forces `use_cache=False` and `past_key_values=None` due to gradient checkpointing incompatibility. This corrupts the autoregressive generation — each step recomputes attention incorrectly through the checkpointing code path.  
**Fix**: Disable gradient checkpointing and restore use_cache before every `generate()` call:
```python
model.gradient_checkpointing_disable()
model.config.use_cache = True
# ... generate ...
model.config.use_cache = False
model.gradient_checkpointing_enable()
```

### Issue: `model.eval()` deadlock
Calling `model.eval()` on a PEFT model with gradient checkpointing active causes a deadlock. Resolution: never call `model.eval()` during evaluation; instead disable gradient checkpointing directly (see above).

## Session 2 — grpo_converge_optimized.py (May 12 2026)

### Goal
Refactor `grpo_converge.py` so the GRPO training engine is reusable and independently testable, while adding memory/performance optimizations from `grpo_gsm8k_optimized.py`.

### Discussion Summary

**Option 1 considered and rejected: use TRL's GRPOTrainer for two models**
- TRL's GRPOTrainer requires a `train_dataset` with a `"prompt"` column — cannot be bypassed
- No public `step(prompts, completions, rewards)` API
- `_compute_loss()` is private and expects fully pre-processed tensor dicts
- Conclusion: TRL's GRPOTrainer is designed for single-model dataset-driven training and cannot cleanly support a two-model game loop

**Option 2 adopted: decompose grpo_converge.py — extract grpo.py, thin game driver**
- Extract the GRPO machinery into a standalone `grpo.py` that knows nothing about the game
- `grpo_converge_optimized.py` imports from `grpo.py` and keeps all game logic verbatim
- `grpo.py` is independently smoke-testable with dummy rollouts (no game needed)

### Optimizations added (from grpo_gsm8k_optimized.py)

| Optimization | grpo_converge.py | grpo_converge_optimized.py |
|---|---|---|
| Optimizer | raw AdamW (full precision) | 8-bit AdamW (bitsandbytes) — ~50% VRAM saving |
| dtype | bfloat16 on capable GPUs | fp16 always on CUDA (A40 is CC8.6, not A100+) |
| Model load | standard | `low_cpu_mem_usage=True` — streams shards to GPU |
| Gradient accumulation | none | configurable via `--gradient_accumulation_steps` |
| use_cache | not explicitly disabled | `model.config.use_cache = False` on load |

### Architecture: grpo.py

Key types/classes:
- `Rollout` dataclass: `prompts`, `completions`, `rewards`, `group_id`
- `compute_completion_logprobs(model, tokenizer, prompts, completions, device)` — chunked forward pass
- `grpo_loss(model, ref_model, tokenizer, rollouts, device, kl_coef)` — standard GRPO loss with within-group reward normalisation
- `GRPOLoop` — wraps model + ref + 8-bit optimizer + gradient accumulation. `step(rollouts) → float | None` (returns loss only when optimizer steps)
- `make_model_and_ref(model_name, use_peft, lora_r, lora_alpha, dtype)` — loads policy + frozen ref with gradient checkpointing

**Two-model usage**: instantiate two `GRPOLoop` objects, one per agent. Each is independent. Coordination (running episodes, splitting results by agent) stays in `grpo_converge_optimized.py`.

### Later in session: shared ref model + Llama 8B

**Problem**: 4 separate 8B fp16 models = 64GB, doesn't fit A40 (48GB).

**Fix**: Since both agents start from the same base, they share the same reference distribution — one frozen ref model suffices for both. This saves ~16GB.

**Change to `grpo.py`**: split `make_model_and_ref` into:
- `make_policy(model_name, use_peft, lora_r, lora_alpha, dtype)` — trainable policy with LoRA + gradient checkpointing
- `make_ref(model_name, dtype)` — frozen reference copy, no checkpointing
- `make_model_and_ref(...)` — kept as convenience wrapper calling both

**Change to `grpo_converge_optimized.py`**: when `model_a == model_b`, loads one shared ref and passes it to both `GRPOLoop` instances. When models differ, loads separate refs.

**Memory with shared ref**: policy_a (16GB) + policy_b (16GB) + shared_ref (16GB) = 48GB — fits A40.
